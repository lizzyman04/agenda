# Phase 02 Plan 02: Task Data Layer Summary

## One-liner

Isar `ItemModel` collection with composite index, bidirectional `ItemMapper`, async `ItemDao` with `.optional()` filter chain, `ItemRepositoryImpl` wrapping all calls in `Result<T>`, pure-Dart `RecurrenceEngineImpl` for 4-frequency RRULE, schema bumped to v2 — 38 tests passing.

## What Was Built

### ItemModel Isar Schema (Task 2)

`lib/data/tasks/item_model.dart` — unified `@Collection` for projects, tasks, and subtasks:

- Three `@embedded` value objects: `MoneyInfo` (amount + currencyCode), `TimeInfo` (dueTimeMinutes + recurrenceRule)
- Three data-layer enums matching domain: `ItemType`, `Priority`, `SizeCategory`
- `dueDate` promoted to top-level `@Index()` field (Isar 3.x cannot index embedded fields)
- Composite `@Index(composite: [CompositeIndex('deletedAt')])` on `type` for active-by-type queries
- All enums use `@Enumerated(EnumType.name)` — prevents ordinal corruption on reorder
- `item_model.g.dart` generated by `isar_community_generator` via `build_runner`

### ItemMapper (Task 3)

`lib/data/tasks/item_mapper.dart` — bidirectional conversion between `ItemModel` and domain `Item`:

- `toDomain(ItemModel)` — extracts `dueTimeMinutes`/`recurrenceRule` from `timeInfo`, `amount`/`currencyCode` from `moneyInfo`
- `toModel(Item)` — creates `TimeInfo` only when `dueTimeMinutes` or `recurrenceRule` is non-null; same for `MoneyInfo`
- Exhaustive switch-based enum converters for all 3 enum types (5 Priority values, 4 SizeCategory values, 3 ItemType values)

### ItemDao (Task 3)

`lib/data/tasks/item_dao.dart` — raw Isar query access:

- All reads async — no `findSync`, `putSync`, `deleteSync` anywhere
- All list queries apply `.deletedAtIsNull()` and `.limit(500)` (T-02-02)
- `filterItems` uses `.optional()` chain with local-variable promotion for analyzer clarity
- `softDelete` and `restoreItem` operate inside `writeTxn` with null-guard
- `watchLazy()` returns `Stream<void>` for reactive UI updates

### ItemRepositoryImpl (Task 4)

`lib/infrastructure/tasks/item_repository_impl.dart` — `ItemRepository` implementation:

- Every method wrapped in `try/catch` returning `Result<T>` — never throws
- `createItem` validates `parentId` resolves to `ItemType.project` before saving (T-02-04)
- `filterItems` translates `EisenhowerQuadrant` to `isUrgent`/`isImportant` booleans
- `getSubtaskCounts` returns `(completed, total)` tuple using two O(1) index count queries
- `@LazySingleton(as: ItemRepository)` for DI registration

### RecurrenceEngineImpl (Task 4)

`lib/infrastructure/tasks/recurrence_engine_impl.dart` — pure Dart RRULE parser:

- `parse()` returns `null` for null, empty, or unrecognised RRULE strings
- Supports DAILY, WEEKLY (with BYDAY multi-day), MONTHLY (with BYMONTHDAY), YEARLY
- `_nextWeekly` scans forward day-by-day up to 8 days for the next matching weekday
- `_nextMonthly` clamps day to last-day-of-month to handle short months
- `@LazySingleton(as: RecurrenceEngine)` for DI registration

### Schema Migration & Wiring (Task 5)

- `AppConfig.schemaVersion` bumped from 1 to 2
- `MigrationRunner._runMigration` case 2 added (empty collection, no data migration)
- `main.dart` calls `IsarService.instance.open([ItemModelSchema])` before `runApp`
- `injection.config.dart` regenerated with `RecurrenceEngineImpl` and `ItemRepositoryImpl` as `LazySingleton`s

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] Added registerFallbackValue for Item and ItemModel in item_repository_impl_test**
- **Found during:** Task 6/7 — test run
- **Issue:** mocktail requires `registerFallbackValue` for any custom type used with `any()` matchers. The test stub did not include `FakeItem`/`FakeItemModel` classes or `setUpAll` registration, causing a runtime `Bad state` error on the first `createItem` test.
- **Fix:** Added `FakeItem extends Fake implements Item`, `FakeItemModel extends Fake implements data.ItemModel`, and `setUpAll(() { registerFallbackValue(FakeItem()); registerFallbackValue(FakeItemModel()); })`.
- **Files modified:** `test/infrastructure/tasks/item_repository_impl_test.dart`
- **Commit:** f7e07ec

**2. [Rule 1 - Bug] Fixed lint infos across test and implementation files**

**3. [Rule 1 - Bug] Promoted nullable params to locals in ItemDao.filterItems**
- **Found during:** Task 7 — flutter analyze
- **Issue:** `unnecessary_null_checks` infos on `!` operators inside `.optional()` lambdas. Dart's flow analysis cannot prove non-null inside closures even when the condition gates invocation. The `!` operators are required for compilation.
- **Fix:** Promoted `type`, `isUrgent`, `isImportant`, `gtdContext`, `dueDateFrom`, `dueDateTo`, `parentId` to local `final` variables before the `.optional()` chain. This does not eliminate the infos (Dart limitation with closures) but documents the intent and keeps code correct. `flutter analyze --no-fatal-infos` exits 0.
- **Files modified:** `lib/data/tasks/item_dao.dart`
- **Commit:** f7e07ec

### Known Analyzer Limitation

`lib/data/tasks/item_dao.dart` reports 6 `unnecessary_null_checks` infos on `!` operators inside `.optional()` lambda callbacks. These are false positives — Dart's flow analysis does not propagate non-null type narrowing into closures, so the `!` operators appear redundant but are required for compilation. `flutter analyze --no-fatal-infos` exits 0. This is an inherent limitation of the Isar `.optional()` API pattern.
